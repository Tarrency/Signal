# AI Coding 协作说明

本项目全程使用Claude Code Sonnet 5实现。

本项目在实现《末班信号站》过程中，使用了多种 AI 辅助能力，但整体原则始终是：**先明确游戏规则与 MVP 边界，再让 AI 协助产出代码、文档与校验结果，最后由人工做取舍和验证**。AI 不是独立完成者，而是提高产出速度、帮助收敛方案和发现问题的工具。

本次主要使用了以下几类 AI 能力：

第一类是**方案设计与需求收敛**。在项目开始阶段，AI 主要用于把模糊题意转化为可执行方案，包括游戏机制、站点结构、调度规则、信号机制、结束条件、排行榜设计等。最初 AI 给出的方案相对更复杂，包含积水等级、多线路、多段位和分数系统；这些输出并没有被完全采纳。原因是笔试题时间有限，复杂机制会抬高实现和验证成本，不利于优先交付一个完整可玩的闭环。最终采纳的是更轻量的版本：2 条线路、3 列车、1 个共享中央信号站、60 秒倒计时、无复杂概率系统、后端权威结算。这一阶段 AI 的主要价值是快速展开多个可选方案，但最终取舍是以“是否更符合 MVP 原则”为标准完成的。

第二类是**代码生成与结构搭建**。AI 被用于生成前后端的基础代码框架，包括 React + Vite 前端结构、Express 后端结构、HTTP API、游戏规则引擎、排行榜存储，以及 README、API 设计文档、游戏设计文档等。这里大部分输出被采纳，但并不是原样接受。例如后端最初曾倾向于拆分更多层级文件，后来被压缩为 `gameEngine.ts`、`routes.ts`、`storage.ts`、`schemas.ts`、`types.ts` 这样的最小结构；前端也从单独的开始页、结果页设计，逐步收敛到更利于笔试交付的轻量页面流转。AI 在这个阶段的价值主要是快速生成可运行初稿，而不是替代架构判断。

第三类是**问题定位与回归修正**。在联调过程中，AI 参与了真实问题的修复：例如 TypeScript 构建失败、NodeNext 与前端模块解析冲突、地图上列车位置没有随状态更新、中央信号站视觉上被画成两个、以及后端一度允许两列车同时冲向中央信号站等。这类输出并非一次成功，而是通过“先运行、看到真实报错、再修正”的方式逐步收敛。这里也有被拒绝或被修正的 AI 输出：例如早期地图展示只适合静态说明，不适合真实共享中央站的玩法；又如 API 文档与实现一度不一致，文档里保留了玩家名输入，但实现已改为固定默认玩家名，这部分后来被修正对齐。换言之，AI 在调试阶段更像一个高效的候选解生成器，而不是绝对正确答案来源。

第四类是**验证辅助**。本项目没有把“AI 说可以”当作完成标准，而是做了真实验证，包括：`npm run build` 构建校验、启动前后端服务后用 `curl` 逐个验证 `POST /api/games`、`GET /api/games/:id`、`POST /api/games/:id/dispatch`、`GET /api/leaderboard` 四个接口，以及通过具体时间窗口验证安全到站、红灯事故、站点占用等关键场景。验证阶段 AI 的用途主要是帮助组织测试步骤、解释失败原因并建议修复顺序，但最终是否通过，以真实命令输出和页面行为为准。

在此基础上补充了**面向核心规则引擎的自动化单元测试**，使用 Vitest（`npm test` 运行），测试文件位于 `test/gameEngine.test.ts`。测试选点的判断依据是“价值密度最高、且天然可确定性验证”的部分：`gameEngine.ts` 是纯函数 + 时间参数注入（`now: Date`）的设计，没有随机数、不依赖真实时钟，因此同样输入必然得到同样输出，非常适合断言。覆盖的场景包括：信号时刻表的取值与左闭右开边界（第 7 秒与第 8 秒切灯）、`getSecondsUntilNextSwitch` 的倒计时与非负兜底、绿灯正常发车、红灯发车抛 `SIGNAL_BLOCKED`、中央站占用抛 `STATION_OCCUPIED`、非可发车状态抛 `INVALID_TRAIN_STATE`、列车到站与红灯抵达中央站判定事故、60 秒结束、三车错峰通关得 S 级与零到站得 D 级，以及对外视图裁剪与排行榜条目生成的前置校验，共 19 个用例。这里 AI 的价值在于快速铺出确定性测试骨架并构造精确的时序输入，而选择“先测引擎、暂不铺开路由与前端组件测试”仍是基于 MVP 时间取舍的人工判断——路由层用 `new Date()` 取当前时间、时序不可注入，其规则分支已在引擎层覆盖，重复投入收益有限。

本次也有明确的**时间取舍**。为了优先完成笔试需要的“可体验闭环”，主动放弃了若干可能更完整但更耗时的方向，例如：用户自定义名称、复杂天气系统、积分体系、多人模式、WebSocket、地图编辑器、全面自动化测试等。即使在后期用户提出补测试需求，方案也倾向于优先补后端规则引擎的最小高价值单测（已落地，见上文第四类），而不是全面铺开路由层与前端组件测试。这个取舍原则同样体现在对 AI 输出的接受上：凡是会显著增加复杂度但对 MVP 核心体验帮助有限的建议，基本都被拒绝或延后。

总结来说，本项目中的 AI 使用方式可以概括为四步：**收敛需求、生成初稿、基于真实运行结果修正、再通过手工验证确认交付质量**。采纳 AI 输出的标准不是“是否看起来高级”，而是“是否更快、更稳地交付一个规则清晰、前后端真实联通、可以现场试玩和验证的 MVP”。这也是本次 AI 协作中最重要的判断依据。
