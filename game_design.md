# 《末班信号站 · 暴雨调度》游戏设计文档

## 一、项目目标

一款 **24 秒倒计时的轻策略调度小游戏**。暴雨将在 24 秒后到达城市，
玩家扮演调度员，必须在暴雨前把多列自动运行的列车安全调回各自终点站。

目标仍是一个**可运行、可解释、可验证的全栈 MVP**：真实前后端 HTTP 通信、
可玩的决策闭环、明确结局、历史最佳成绩榜。

---

## 二、核心设定

暴雨 24 秒后到达。3 列末班空车已自动发车，前方有开局即损坏的路段（红色）。
若不加干预，列车会驶入损坏路段、或在共享路段相撞。
调度员的职责：**在分岔口点击方向箭头变道**，让尽可能多的空车安全回到终点站（次日始发点）、零事故。

---

## 三、地图设计（左→右格状曲线网络）

采用**严格向右推进的格状图**：始发端(col0) → A/B/C/D/E 五层枢纽 → 终点站(col6)，
共 20 节点 / 33 条路段。路段以**曲线（二次贝塞尔）**绘制。

- **所有路段只向右**（`from` 的 x 恒小于 `to`），列车只会越走越靠近终点。
- **每个分岔口最多 2 个前进方向：右上 / 右下**（`forwardNeighborsOf` 按 x 过滤），
  不存在向左上/左下的变道选择。
- 多条并行/交叉路径天然存在共享路段；不同列车走不同替代路时共享的路段也不同。
- 每条路段有固定通过秒数与 `curve` 弯曲量；前端 `geometry.ts` 用同一套贝塞尔公式
  绘制路段、列车运动轨迹与方向箭头，三者严格对齐。
- 列车默认路线由 Dijkstra 求 始发端→终点 最短路得到。

---

## 四、列车设计

共 3 列**末班空车**：乘客已散，需运回各自终点站过夜；这些终点站也是次日早班的发车起点。
各车有独立始发端与终点站，开局即自动运行：

- **T1**：甲始 DA → 甲站 TA
- **T2**：乙始 DB → 乙站 TB
- **T3**：丙始 DC → 丙站 TC

三车默认最短路会共用部分枢纽路段；开局损坏落在默认路线上，逼玩家在分岔口变道，
不同的变道选择又会改变彼此的共享路段，从而带来相撞风险的取舍。

---

## 五、时间与运行模型

- 暴雨倒计时 **24 秒**（挂钟推进、不冻结；到达即结束）。干净跑完约 16~18s，
  留约 6~8s 余量：暂停(3s)有感、事故检修(6s)一次会明显挤压到站窗口，
  时间因此成为真资源。
- 服务端为权威模拟，**逐秒离散推进**；每秒顺序为：先推进列车 → 再结算雷击 → 再检测相撞。
- 运行模型：列车开局即自动运行；抵达节点后**同一秒**驶入规划路线的下一段，
  在路段上按该段秒数前进，走完抵达下一节点。
- 前端轮询 500ms + CSS 过渡，让列车 marker 沿曲线平滑移动。

---

## 六、损坏道路与雷击

- **开局即损坏**：新建对局时按种子随机选 **3 段**路损坏。它们**落在列车默认路线上**
  （制造变道动机），但**不在任一列车的第一段**（留反应时间），
  且保证每列车仍存在避开损坏抵达终点的可行路线。
- **中途雷击**：第 **6 / 14 / 22 秒**各再随机劈坏一条当时未被占用、也未走过的路段
  （都落在列车仍在路上的窗口）。若某秒有列车即将抵达下一节点（路段剩余 ≤1 秒），
  雷击会推迟到列车已驶入下一段、尚未临近到站时再结算，避免「到站瞬间路段变红」。
  选段时还会避开各车当前路段及其下一程出站路段。
- **已走过的路段不再被劈坏**（`traveledEdgeIds`）。
- 损坏路段染**统一红色**并脉动，进入 `damagedEdgeIds`；地图绘制时红色路段在路线底图之上，避免被盖住。
- 全程**种子随机**（mulberry32，seed 存入对局），保证同一对局可复现、可验证。

---

## 七、事故与检修（列车不报废）

事故有两种，均为确定性判定：

1. **相撞**：同一秒有两列车占用同一条公共路段。
2. **驶入损坏**：列车沿已损坏的红色路段**实际移动**至少 1 秒。仅路过该路段端点节点、或刚挂上该路段但仍在端点（elapsed=0）不算驶入。

事故发生 → **全体列车检修停滞 6 秒**（倒计时照走）；6 秒后：

- 清空并**重 roll 2 段**新的随机损坏路段（保证各车从当前位置仍可达终点）；
- **顶替（轮空）一次**定时雷击。

列车**不报废**，检修后继续沿路线运行。相撞用 `collidedPairs` 去重，避免检修后原地反复触发。
`accidentCount` 计入结果的「事故数」。

页面提示只写原因类型与车次，例如 `T1 驶入损坏路段`、`T1、T2 相撞`，**不展示具体路段名**。

> 末端**选错终点站**是例外：列车驶入非目标终点站后卡住（`stranded`），
> 记为未到站，但不触发检修。

## 八、玩家操作：地图变道箭头 + 全体暂停

**只操作地图**，两种动作：

**变道箭头**

- 列车**即将抵达的分岔口**显示各前进方向的**方向箭头**（该车颜色）；
  地图为严格左→右格状网，箭头只指右上 / 右下，最多 2 个选择。
- 列车正驶入该分岔口所在路段时箭头**亮起**（发光脉动），否则暗淡。
- **hover** 箭头 → 高亮预览变道后完整路线；**点击** → 从该方向重算避开损坏的最短路线。
- 指向损坏路段的箭头呈红色、不可点击。
- 接口：`POST /api/games/:id/reroute { trainId, viaNodeId }`。

**全体暂停**

- 地图内一个「全体暂停」按钮：按一次 → **所有列车停滞 3 秒**（倒计时继续），用于错峰避撞。
- 停滞期间列车冻结、不显示变道箭头；3 秒后自动恢复。停滞中再按被拒绝。
- 反馈只使用地图上的黄色停滞横幅，**不再额外弹绿色 toast**。
- 接口：`POST /api/games/:id/pause`。

全局停滞（暂停/检修）期间列车与雷击都冻结，仅倒计时推进。

## 九、页面与反馈（极简）

页面**只保留倒计时模块与调度地图**，不展示调度日志、列车卡片、暴雨面板。

- **顶部**：暴雨倒计时 + 到站/事故/暂停/在途计数。
  - 倒计时 ≤10s：顶部模块进入红色紧张态。
  - 本局结束：顶部模块暴雨雨幕特效，标题前显示 **⏸**，并出现「查看结果」按钮。
- **SVG 地图**：
  - 曲线路段（虚线）；列车经过时该段染成该车颜色流动；
  - 损坏路段统一红色脉动并置于路线底图之上；
  - 每列车**剩余规划路线**以低透明度各车颜色铺底（避免已走路段与他车行驶段串色）；
  - 分岔口方向箭头（临近亮起 / hover 预览 / 点击变道）；
  - 列车 marker 沿曲线平滑移动。
- **停滞横幅**：
  - 暂停：`⏸ 全体暂停 · 列车停滞 Ns（倒计时继续）`
  - 检修：`⚠ {原因} · 全体检修 Ns（倒计时继续）`（原因为驶入损坏或哪几辆车相撞）
- **变道提示**：点箭头成功后地图上方弹出 toast（真正变道显示「已变更车道」，
  点到当前方向显示「保持原路线」），约 1.6s 消失；变道后清除 hover 预览。
- **到站不自动跳转**：列车保留在终点站；暴雨到达后由玩家点击「查看结果」进入结果页。
- **结果页**：展示评级、指标与历史榜；提供「返回首页」（回介绍页）与「再来一局」。

## 十、结果与评级

- **S**：3 车全到、0 事故、0 暂停（全靠变道零失误）
- **A**：3 车全到、0 事故
- **B**：3 车全到（有事故/暂停）
- **C**：2 车到站
- **D**：其余（≤1 车到站）

终态：`arrived`（到站）/ `stranded`（误入他站）；暴雨到达时仍在途记为未到站。
事故不再是列车终态，而是累计的检修次数。

---

## 十一、历史最佳成绩榜

服务端持久化（`data/leaderboard.json`）。排名规则：到站多 → 事故少 → 用时短 → 记录早。
服务端启动时读入内存缓存；清空文件后需重启后端才会反映为空榜。

---

## 十二、API 概览

- `POST /api/games`：新建对局
- `GET  /api/games/:id`：获取（并推进）对局状态
- `POST /api/games/:id/reroute`：变道 `{ trainId, viaNodeId }`
- `POST /api/games/:id/pause`：全体暂停 3 秒
- `GET  /api/leaderboard`：历史成绩榜

---

## 十三、可复现与可验证

- 固定 seed → 开局损坏、雷击与时序可复现。
- 服务端保留全量日志便于核对（页面不展示）。
- 事故原因确定：相撞或驶入损坏路段；无黑盒概率。
- 单元测试覆盖：寻路、开局损坏约束、自动前进、驶入损坏判定、暂停、方向箭头决策、变道避障、雷击时机与保护、评级、视图序列化。
